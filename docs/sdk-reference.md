# SDK Reference

Non-normative. The testable rules are [spec/sdk.md](spec/sdk.md) (`REQ-SDK-*`). Implementation: `sdk/*.go`.
See [concepts/workflow.md](concepts/workflow.md) for why the SDK exists at all — a workflow authored here
compiles to the exact same `domain.Workflow` a YAML file produces, and hashes identically (PRIN-03/04).

The SDK is the same Go module as the engine, embedded directly — no subprocess, no serialization boundary
at authoring time. Use it when a workflow's shape is more naturally expressed as code than YAML (a fan-out
whose worker count depends on a variable, a workflow generated by a script), not as the default way to
author one — YAML remains the canonical form every example under `examples/` uses.

## Building a graph — `sdk.New(...).Build()`

`Builder` tracks a **frontier**: the set of nodes a subsequent step depends on. `Worker`/`Tool` add one node
after the frontier (the common linear case); `Parallel` fans out into several; `Merge` fans a multi-node
frontier back into one.

```go
wf, err := sdk.New("pr-review", "1.0.0").
    Budget(budget).
    Parallel(
        sdk.Worker("review-security", secReviewer),
        sdk.Worker("review-style", styleReviewer),
        sdk.Worker("review-correctness", correctReviewer),
    ).
    Merge("fix", fixer).
    Tool("test", testRun).
    Tool("commit", commit).
    Build()
```

This is the flagship's exact shape (3 reviewers → fixer → test → commit) — compare against
[examples/pr-review/workflow.yaml](../examples/pr-review/workflow.yaml).

| Method | Effect |
|---|---|
| `sdk.New(id, version)` | starts a builder |
| `.Budget(domain.Budget)` | sets the required Budget field |
| `.Worker(nodeID, w)` | adds a worker-backed node depending on the current frontier |
| `.Tool(nodeID, tc)` | adds a tool-backed node depending on the current frontier |
| `.Parallel(specs...)` | adds several nodes each depending on the current frontier; frontier becomes all of them |
| `.Merge(nodeID, w, from...)` | adds a worker-backed node fanning in from `from` (or the whole frontier if omitted) |
| `.Build()` | validates and returns the compiled `*sdk.Workflow`, or the first error recorded along the way |

`sdk.Worker(nodeID, w)` / `sdk.Tool(nodeID, tc)` (package-level, not `*Builder` methods) build a `Spec` for
use inside `Parallel` — the node id and the Worker are separate; the graph node references the Worker by
`"id@version"`, matching how a YAML `node.worker` field works. A duplicate node id anywhere in the chain is
recorded as the builder's error and surfaced at `Build()`, not panicked immediately — so a long builder
chain fails once, at the end, with one clear message rather than a stack trace mid-chain.

## Running it — `(*Workflow).Run`

```go
exec, err := wf.Run(ctx, sdk.RunOptions{})   // zero value works: ".workflow", default providers, no tools
if err != nil { /* ... */ }

for ev := range exec.Events() {              // live stream — best-effort, see below
    fmt.Println(ev.Type, ev.NodeID)
}
result, err := exec.Wait()                   // blocks for the final result
```

`RunOptions` fields, all optional:

| Field | Default | Meaning |
|---|---|---|
| `Workspace` | `".workflow"` | state directory (executions, artifacts, cache) |
| `Providers` | `providers.Default()` | resolves a Worker's provider name to an implementation (openai + anthropic, keys from env) |
| `Tools` | empty registry | sandboxed tool registry for tool-backed nodes — wire built-ins yourself if the workflow has any |
| `Concurrency` | engine default | max nodes to run in parallel |
| `Cache` | on | node cache mode |

`Run` returns immediately with an `*Execution`; the run itself happens on a background goroutine, the same
`engine.Scheduler` `wee run` uses underneath.

- `Execution.Events() <-chan domain.Event` — a **best-effort** live view: if the consumer falls behind the
  256-event buffer, events are dropped from this channel, not from the record. The event log is always the
  complete, ordered source of truth (PRIN-02) — readable via `core/replay` regardless of whether anything
  ever read the channel.
- `Execution.Wait() (*engine.Result, error)` — blocks until the run finishes. Independent of `Events()`:
  calling only `Wait()` and never draining the channel is fine.

## Reading a result — `sdk.Artifact[T]`

```go
type reviewResult struct {
    Verdict string   `json:"verdict"`
    Issues  []string `json:"issues"`
}

out, err := sdk.Artifact[reviewResult](exec, "review-security")
```

`Artifact[T]` (REQ-SDK-02) waits for the run, finds the named node's stored output by hash, and unmarshals
it into `T`. Because the artifact was already validated against the node's Contract before it was ever
stored (REQ-CONTRACT-01), `T` only needs to describe the shape you want in Go — the schema enforcement
already happened. A node that failed, was skipped, or produced no artifact is a returned `error`, not a
zero value silently standing in for "didn't run."
